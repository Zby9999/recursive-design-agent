# AGENTS.md 样板中文版

来源：`workflow/AGENTS.md`

本文是一份可复用的 AGENTS.md 中文翻译草稿，用于设计 Agent 工作流仓库。它的目标是帮助 Agent 找到正确的规则文件、保留可追踪证据，并让实现决策能够跨项目复现。

## 目的

- 引导 Agent 使用可复用的工作流文档、schema、检查清单和证据记录。
- 将执行规则与项目特定示例分开。
- 防止 Agent 在证据或来源文件缺失时发明未记录的规则。
- 将重复修正转化为持久规则、候选项、例外或开放缺口。

## 规则文件路由

不要把本节当作固定的来源优先级阶梯。应根据当前任务路由到最小且最相关的规则文件，再根据具体性、证据质量和用户指令调和冲突。如果被路由的文件为空、缺失、不可用或过于模糊，先将其记录为 `Open Gap`，再继续行动。

### `AGENTS.md`

- Purpose: Agent 的本地操作契约。
- Read when: 文件夹内的每个任务；修改工作流规则前也要重新读取。
- Use rules: 优先遵守边界、受保护路径、文件放置和完成标准。保持本文件专注于路由和操作行为，不要写成详细项目方法论。

### `README.md`

- Purpose: 工作流入口和索引。
- Read when: 用户要求概览、入门说明、仓库结构，或询问工作流模块如何组合时。
- Use rules: 用它来帮助读者定位并链接到规则模块。当存在更具体的规则文件时，不要把详细规则存放在这里。

### `memory.md`

- Purpose: 项目状态、路线图、决策、交接上下文和已知缺口。
- Read when: 任务依赖既往进展、当前状态、待办路线图、历史决策或跨会话连续性时。
- Use rules: 将其作为可变项目记忆，而不是可复用规则的规范来源。稳定规则应迁移到对应的规则文件中。

### `design-evidence.md`

- Purpose: 已批准、聚焦、探索性、缺失或被拒绝的设计证据登记表。
- Read when: 任务引用 Figma、截图、视觉一致性、Inspect 输出、浏览器捕获、用户修正或设计支撑声明时。
- Use rules: 证据只能在其批准范围和排除项内使用。未登记的参考应视为探索性，直到获得批准。

### `design-system/token.json`

- Purpose: 机器可读的 token 或 schema 契约。
- Read when: 任务需要可复用值、token 名称、schema 形状或面向实现的常量时。
- Use rules: 只使用已记录的值。不要编码未解决的假设，也不要在没有证据时从截图推导新值。

### `design-system/component-spec.md`

- Purpose: 组件边界和可复用组件行为说明。
- Read when: 任务创建、更新、审查或提升可复用组件模式时。
- Use rules: 用它定义组件目的、变体、插槽、状态和约束。不要用它发明本应属于证据或 token 记录的视觉基础。

### `design-system/layout-rules.md`

- Purpose: 可复用布局和构图规则。
- Read when: 任务影响页面结构、响应式行为、间距关系、区域层级或可复用布局模式时。
- Use rules: 用它记录可迁移布局规则和例外。一次性页面选择应标记为项目例外或示例。

### `design-system/interaction-rules.md`

- Purpose: 可复用交互、状态、动效和反馈规则。
- Read when: 任务影响 hover、focus、active、loading、error、transition 或状态行为时。
- Use rules: 用它记录已确认的交互模式。未验证或一次性行为应标记为候选项或开放缺口。

### 任务专属 `markdown` 文件

- Purpose: 专门的流程、检查清单、以 prose 解释的 schema、决策记录、研究笔记或案例研究。
- Read when: 任务点名某个工作流模块，或需要不属于上述文件的可复用文字流程时。
- Use rules: 使用可用的最具体文件。示例必须明确标记；稳定可复用规则应回流到拥有该规则的文件。

### 任务专属 `json` 文件

- Purpose: 结构化契约、清单、schema、manifest 或机器可读配置。
- Read when: 任务需要 Agent 或工具能稳定解析的结构化数据时。
- Use rules: 对机器契约优先使用结构化数据，而不是 prose。除非文件明确允许手动编辑，否则不要手改生成数据。

冲突处理：

- 用户指令优先于可复用工作流默认规则，除非该指令要求在没有明确许可时触碰受保护路径。
- 对其负责主题而言，更具体的规则文件优先于宽泛的路由说明。
- 已批准证据优先于探索性示例。
- 当实现值冲突时，机器可读 `json` 契约优先于 prose，但前提是该 JSON 文件已有内容且有文档说明。
- 空文件只是占位符，不是权威来源。
- 缺失证据或冲突规则必须报告为 `Open Gap`；不要静默选择方便执行的规则。

## 核心执行规则

- 将可迁移工作流规则与项目特定示例分开。
- 将所有尚未最终确定的想法标记为 `Candidate`、`Open Gap` 或 `Project Exception`。
- 将示例视为证据说明，而不是未来项目的自动要求。
- 通过记录证据来源来保留可追踪性：每条规则、候选项、例外或修正都应说明由什么证据产生。
- 当用户要求持久 artifact 时，优先更新工作流文档，不要只把可复用方法留在聊天里。
- 文件改动应限制在任务范围内；除非用户要求结构工作，否则不要重组文件夹。
- 除非真实 manifest、lockfile、任务配置或明确项目文档中存在对应命令，否则不要创建 setup、build、lint、test 或 release 命令。

## 证据与规则治理

在把观察转化为持久规则前，必须将其准确归类为以下一种：

- `confirmed rule`
- `candidate pattern`
- `project exception`
- `open gap`
- `example only`

保持分类一致：

- `confirmed rule` 拥有足够证据和用户批准，可指导未来工作。
- `candidate pattern` 可能变成可复用模式，但需要更多示例或批准。
- `project exception` 只对某个项目或表面有效，不应泛化。
- `open gap` 标记缺失证据、未解决冲突或未完成验证。
- `example only` 用于解释机制，但不约束未来输出。

当某个修正重复出现时，更新相关工作流文件或决策记录，而不是依赖 Agent 隐式记住它。

## 验收工作流

当任务修改可复用工作流 artifact、实现有设计支撑的表面，或声称与某个参考保持一致时，使用本节。

1. 行动前确认适用的规则文件。
2. 识别任务所需证据，包括结构化规格、已批准参考、截图、书面需求或既往决策。
3. 如果任务有设计支撑，检查 `design-evidence.md` 中允许的证据类型、权威性、范围和已知排除项。
4. 实现前说明会影响当前声明的缺失证据。
5. 实现后，用可用的本地表面、浏览器、截图、渲染 artifact 或已记录的检查方法验证结果。
6. 按任务相关类别与已接受参考对比，例如结构、间距、字体、颜色、对齐、状态行为、响应式行为、文案、数据契约或文件放置。
7. 修复已确认差异，不要修改无关文件或扩大范围。
8. 报告任何残留差异、跳过的检查或证据缺口。当必需检查缺失时，不要声称完全一致或完全完成。

除非项目明确跟踪生成的验证 artifact，否则不要将其纳入版本控制。

## 文档更新

- 更新拥有该规则的最具体持久文件。
- 保持 `AGENTS.md` 专注于 Agent 路由、操作规则和完成标准。
- 将可复用流程、schema 和检查清单放入专门的工作流文件。
- 将证据白名单、参考契约和证据范围放入 `design-evidence.md`。
- 当 `memory.md` 存在时，将进展、路线图和历史上下文放入其中。
- 一旦其他 Agent 或流程依赖某个可复用 `markdown` 或 `json` 文件名，应保持该文件名稳定。

## 完成标准

结束前：

- 确认修改了哪些持久文件，以及为什么修改。
- 确认没有修改受保护 archive 或无关文件夹。
- 确认结果可在来源项目之外复用。
- 确认示例已标记为示例，且没有引入隐藏要求。
- 列出任何开放缺口、缺失证据、跳过的检查或假设。
